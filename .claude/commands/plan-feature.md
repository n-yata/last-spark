# 新機能の要求企画 (planモード)

このコマンドは新機能の**要求を対話的に練り**、内容が固まったら
`.steering/[日付]-[タイトル]/requirements.md` として記録します。
生成した `requirements.md` は、後続の `/add-feature` がそのまま要求として読み込み、
**再生成せずに尊重して** design.md / tasklist.md の生成・実装へ進みます。

```
[/plan-feature] planモードで構想 → requirements.md 作成 → 停止
                                          ↓ ユーザーが内容確認のうえ手動で起動
[/add-feature]  requirements.md を尊重（再生成しない）
                → design.md / tasklist.md 生成 → 実装
```

**引数:** なし。このコマンドは**常に引数なしで起動する**。テーマは引数で受け取らず、
起動後に必ずステップ1の**対話**で決める。たとえユーザーが文字列を添えても、それは
対話の出発点としてのみ扱い、ディレクトリ名やタイトルは対話で合意したものを使う。

**重要:** このコマンドは **requirements.md の作成までで停止する**。design.md / tasklist.md の
生成や実装には**進まない**。それらは `/add-feature` の責務。
requirements.md 作成後に `/add-feature` を自動起動してはならない。

---

## ステップ1: 要求を練る（planモード）

### 1-A. まず問いかける（初回レスポンス／ゼロ読込で即返す）

> **重要（レスポンス速度）:** 初回レスポンスは**ファイル読込・`Grep`/`Glob` を一切行わずに**返すこと。
> 起動直後にやることは「テーマを問いかける」ことだけ。`CLAUDE.md` や `docs/` の読込、関連実装の
> 調査は **1-C（テーマ確定後）まで絶対に行わない**。ここで前倒しに調査すると初回応答が大幅に遅くなる。

1. **テーマをヒアリングする（最初に必ず、これだけを行う）**
  - まずユーザーに「どんな機能を考えたいか」を問いかける。引数からテーマを決めつけない。
  - 当てがある場合はそれを起点に深掘りする。なければ、（記憶している範囲の）プロジェクトの現状を
    踏まえて候補を軽く提示してよいが、**そのためにファイルを読みに行かない**。
  - ここで一旦ユーザーの応答を待つ。`EnterPlanMode` を含むツール呼び出しもこの応答の後で構わない。

### 1-B. planモードに入る

2. **planモードに入る。** 以降、確定前はファイルを書かず、ユーザーと対話して構想を固める。

### 1-C. テーマ確定後に調査する（初回レスポンス後）

3. **プロジェクト方針の把握**（**1-A でテーマが固まってから**実施。初回レスポンス前には読み込まない）
  - 必要に応じて `CLAUDE.md` および `docs/product-requirements.md` / `docs/functional-design.md` /
    `docs/architecture.md` を読み、プロジェクトの北極星と整合する要求か確認する。
  - **確定したテーマに関連する**既存実装のみ `Grep` / `Glob` で軽く調査し、実現可能性の感触を掴む。
    テーマ未確定のまま広く調査しない（的が絞れず無駄打ちになる）。

4. **対話で以下を引き出す**（不明点は仮置きせず、ユーザーに確認する）
  - 機能のタイトル（短く識別できる名前）
  - 背景（なぜ作るのか、どんな課題を解くのか）
  - 実装対象の機能（What。ユーザー体験としてどうなるか／実現したい振る舞い）
  - スコープ（含む / 含まない）
  - 受け入れ条件（検証可能な形で）
  - 成功指標（定量・定性、あれば）
  - 未決事項（実装フェーズで決める点があれば）

5. **要求が固まったら、ユーザーに合意を取る。**
  - 「この内容で requirements.md にまとめてよいか？」を確認し、承認を得てからステップ2へ。

## ステップ2: requirements.md の生成

1. **ステアリングディレクトリを決める**
  - 現在の日付を `YYYYMMDD` 形式で取得する。
  - **ステップ1の対話で合意したタイトル**を簡潔な名前にし、`.steering/[YYYYMMDD]-[タイトル]/` を決定して作成する。
  - **`/add-feature` と同じ命名規約**にすること（add-feature が同じディレクトリを再利用できるようにするため）。

2. **requirements.md を作成する**
  - `.claude/skills/steering/templates/requirements.md` のテンプレートに準拠し、
    ステップ1で固めた内容を埋めて `.steering/[YYYYMMDD]-[タイトル]/requirements.md` に書き出す。
  - テンプレートの章立て（概要 / 背景 / 実装対象の機能 / 受け入れ条件 / 成功指標 / スコープ外 / 参照ドキュメント）を踏襲する。
  - 未決事項がある場合のみ、末尾に `## 未決事項` セクションを任意で追加し、
    実装フェーズで判断・確認すべき点を箇条書きで残す（add-feature の設計フェーズが拾えるようにするため）。

  > ※ ここで作る requirements.md は `/add-feature` が**再生成せずそのまま使う**正式な要求書。
  > 「ざっくり」ではなく、受け入れ条件は検証可能な粒度まで詰めること。

3. **ユーザーに報告し、次の一手を案内する**
  ```
  requirements.md を作成しました: .steering/[YYYYMMDD]-[タイトル]/requirements.md
  内容を確認のうえ、続けて実装する場合は次を実行してください:
      /add-feature [タイトル]
  add-feature は同じステアリングディレクトリの requirements.md を再生成せず尊重し、
  design.md / tasklist.md の生成・実装へ進みます。
  ```

4. **ここで停止する。** `/add-feature` を自動起動しない。

## 完了条件

- `.steering/[日付]-[タイトル]/requirements.md` が作成されている。
- requirements.md が steering テンプレートの章立てに準拠している。
- ディレクトリ名が `/add-feature` と同じ命名規約になっている。
- ユーザーに `/add-feature [タイトル]` での続行を案内している。
- design.md / tasklist.md の生成・実装には踏み込んでいない（requirements.md までで停止）。
