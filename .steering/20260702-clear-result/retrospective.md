## 学んだこと

**技術的な学び**:
- **必須フィールドと任意フィールドでセーブ拡張のコストが違う**: `busterMode` / `vibration`(settings 内の必須)はバージョン繰り上げ+マイグレーション+全リテラル洗い出しが必要だったが、今回の `bestRank`(トップレベル任意)は `bestTimeMs` と同方式で version 据え置き・マイグレーション不要・後方互換(旧コードは未知フィールドを無視)で済んだ。**セーブ拡張はまず「任意フィールドで表現できないか」を検討する価値がある**。
- **Phaser Scene のフィールド初期化子はシーン再入で走らない**。集計・状態フィールドを足すときは必ず `create()` のリセット節へも追加する(GameScene には既にリセット節があるので、そこに足すだけ)。
- **「保存前に読む」が必要な判定は保存 API と分離する**: NEW RECORD 判定は `markStageCleared` の前に既存ベストを読む必要がある。保存メソッドに判定を混ぜず、純粋関数 `isNewRecord(prev, cur)` にして呼び出し順を呼び出し側の責務にした。stage6(保存が GameScene 側)のような経路違いにも `newRecord` フラグの受け渡しで対応できた。
- **実プレイ検証はボス HP 操作で短縮できる**: テレポート(ボストリガー到達)+ boss.hp=1 + 実弾トドメで、combat 経路(onBossDefeated → handleClear → ClearScene)を数十秒で通せる。道中の kills 加算も実弾撃破で確認し、「配線が本番経路で動く」ことを担保した。

**プロセス上の改善点**:
- レイヤー依存ルール(repository-structure.md の依存可能/禁止)を、import を書く**前**に確認する習慣が要る。今回は書いた直後に気づいて手戻りが小さかったが、設計段階で「このモジュールは誰から参照されるか」を書き出せば防げた。
- 前作業(stage-select-cards)で用意した受け皿(`StageCardModel` の拡張余地・`rankColor` の共有方針)がそのまま効き、カード側の変更は数行で済んだ。「次の作業の受け皿を1つ用意しておく」進め方は継続したい。

## 次回への改善提案
- セーブ拡張の判断基準をチェックリスト化: ①任意フィールドで表現できるか(できれば version 据え置き) ②必須なら version 繰り上げ+migrate+全リテラル洗い出し(20260702-input-feel の⑥⑦⑧参照)。
- ランクの閾値(`RANK.aDamageRatio`)・演出時間(カウントアップ 900ms)は実機の体感で調整余地あり。特に hard モードは被ダメ倍率が掛かるため S/A が取りにくい。難易度別の閾値調整が要るかはプレイヤーの反応を見て判断する。
- GameOver 画面にも戦績(どこまで進んだか・撃破数)を出すと再挑戦の動機づけになる(今回スコープ外、候補として残す)。
