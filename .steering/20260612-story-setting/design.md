# 設計書

## アーキテクチャ概要

今回の変更はすべてドキュメント層のみ。ソースコード (`src/`) への変更はなし。

```
docs/
├── story.md          ← 新規作成（ストーリー設定書）
├── product-requirements.md  ← 一部更新（固有名詞追記）
└── glossary.md       ← 一部更新（RAY / ECLIPSE / 科学者 追記、既存エントリ更新）
```

## コンポーネント設計

### 1. `docs/story.md`（新規作成）

**責務**:
- LAST SPARK のストーリー設定を一元管理する北極星ドキュメント
- 今後のステージ追加・ログテキスト作成・アセット方針の判断基準

**章立て**:
1. 概要（1文のエレベーターピッチ）
2. 世界設定（舞台・現状・視覚的トーン）
3. 登場人物（RAY / ECLIPSE / 科学者 / 囚われた人間）
4. 物語の骨格（発端→展開→対決→結末方向性）
5. ストーリーテリング方針（環境ストーリーテリング・ログ断片・レイヤー構造）
6. 未決事項

**実装の要点**:
- requirements.md の内容を忠実に展開する（再解釈・追加設定を加えない）
- 未決事項（囚われた人間の名前・科学者の名前・結末詳細）は明示的に未定義として残す
- 今後の意思決定で参照しやすいよう、各章に「開発上の指針」を付記する

### 2. `docs/product-requirements.md`（一部更新）

**更新箇所**:
- プロダクトコンセプトの「退廃の中の希望」説明文に RAY / ECLIPSE の固有名詞を追記
- 世界観・コンセプト記述を `story.md` の内容と整合させる

**実装の要点**:
- 既存の受け入れ条件・機能要件・非機能要件には一切触れない
- 固有名詞の追記のみ。既存の文章構造を変えない

### 3. `docs/glossary.md`（一部更新）

**更新箇所**:
- 既存エントリ「最後のロボット」→ RAY を正式名称として追記・参照
- 既存エントリ「管理AI」→ ECLIPSE を正式名称として追記・参照
- 新規エントリ「RAY」を追加（ドメイン用語セクション）
- 新規エントリ「ECLIPSE」を追加（ドメイン用語セクション）
- 新規エントリ「科学者（RAYの創造者）」を追加（ドメイン用語セクション）
- 索引の A-Z セクションに RAY / ECLIPSE を追加

**実装の要点**:
- 既存エントリの削除・大幅改変はしない（追記のみ）
- 用語の書式は既存エントリに合わせる（定義・説明・関連用語・英語表記）

## データフロー

### ドキュメント間の参照関係

```
docs/story.md  ←  docs/product-requirements.md（プロダクトコンセプトから参照）
docs/story.md  ←  docs/glossary.md（各用語の詳細から参照）
docs/story.md  ←  将来のステージ設計・ログテキスト作成（判断基準として参照）
```

## テスト戦略

### ドキュメント品質チェック（手動）
- `story.md` の全章立てが requirements.md の受け入れ条件を満たしているか
- `glossary.md` の新エントリが既存フォーマットと一致しているか
- `product-requirements.md` の更新が既存文章を壊していないか

### 自動テスト
- ソースコード変更なし → `npm test` / `npm run lint` / `npm run typecheck` は影響なし

## ディレクトリ構造

```
docs/
├── story.md          ← 新規作成
├── product-requirements.md  ← 更新（プロダクトコンセプト欄の固有名詞追記）
└── glossary.md       ← 更新（RAY / ECLIPSE / 科学者 追記）
```

## 実装の順序

1. `docs/story.md` を新規作成（メインの成果物）
2. `docs/product-requirements.md` を更新（固有名詞追記）
3. `docs/glossary.md` を更新（新エントリ追加・既存エントリ更新）
4. 受け入れ条件の確認
5. `npm test` / `npm run lint` / `npm run typecheck` の実行

## セキュリティ考慮事項

- ドキュメントのみの変更のため、セキュリティリスクなし

## パフォーマンス考慮事項

- ドキュメントのみの変更のため、パフォーマンスへの影響なし

## 将来の拡張性

- 科学者の名前・囚われた人間の詳細設定は次フェーズで `story.md` に追記する
