# 実装後の振り返り

## 作業概要
タイトル画面のステージセレクトをテキスト縦並びからカード式グリッドへ格上げした。各カードに背景テーマの手続き生成ミニプレビュー・CLEARバッジ・ベストタイム・段階解放(LOCKED)を表示する。解放判定は「直前ステージを一度でもクリアした記録(clearedStages または周回で消えない bestTimeMs)」で、New Game+ 後も到達済みステージを選び直せる。

## 実装完了日
2026-07-02

## 計画と実績の差分

**計画と異なった点**:
- `formatBestTime` は設計時 `m:ss.d` としたが、既存 TitleScene / ClearScene の `m:ss` と表示形式を揃えることを優先して `m:ss` にした。併せて両シーンの重複 private `formatTime` を共通関数へ集約した。
- シルエット列がカード右端からはみ出す不具合をスクリーンショット確認で発見。`generateSilhouetteColumns` は worldWidth+step まで列を敷く仕様(実ステージではカメラが切る)のため、Graphics 直描きでは自前クリップが必要だった。
- モバイル相当(dpr=2, 667x375)でカード名と BEST タイムが数px重なることを実測で発見し、プレビュー高を 52%→46% に調整した。
- **ポーズメニューのステージ移動パネルが解放ルールの素通り経路になる**ことに実装後に気づき、`isStageUnlocked` を共用してロックを適用した(未解放は [LOCKED] 非活性行)。ロック機能はすべての選択経路に同時適用しないと意味がない。

**新たに必要になったタスク**:
- 上記4点(表示形式の統一と重複解消、右端クリップ、プレビュー高調整、ポーズ経路のロック適用)。

## 学んだこと

**技術的な学び**:
- **既存の背景テーマ資産(config の純データ + 決定論生成)はUIプレビューへそのまま転用できる**。`stageBackground.ts` の skyTop/skyBottom/accent/layers/seed と `generateSilhouetteColumns` を縮小描画するだけで、アセット追加ゼロで6ステージの視覚差別化ができた。純データ+決定論生成で設計しておくと再利用が効く。
- **Graphics 直描きのプレビューは「カメラが切ってくれない」**。ワールド描画用の生成ロジック(worldWidth を超えて敷く等)を UI に流用するときは、描画側でのクリップ(範囲外スキップ + 幅クランプ)が必須。
- **Phaser の `fillGradientStyle` は WebGL 限定**。Canvas フォールバックと見た目を揃えるには `lerpColor` の帯分割で描く(既存の色補間関数がそのまま使えた)。
- **Playwright 検証の罠(新規)**: dpr=1 の headless でビューポートを変更すると、Phaser 側は `scale.resize` に追従するが canvas の inline style(CSS寸法)が古いまま残り、クリック座標がずれて全画面スタートゾーンに吸われる。原因は Phaser ScaleManager が **zoom===1 のとき resize() で canvas.style を書き換えない**こと(dpr>1 では zoom≠1 なので起きない)。検証は `deviceScaleFactor: 2` のモバイルエミュレーションコンテキストで行うと実機条件にも近く安定する。この問題自体は既存の潜在バグとして別タスクに切り出した。
- **UIの検証は「テキスト境界の画面内判定」と「要素同士の重なり判定」を数値でやる**とスクリーンショット目視より確実。`getBounds()` の AABB 交差でカード名×BESTの重なりを機械検出できた。

**プロセス上の改善点**:
- ロック(制限系)機能は「他の経路で同じ操作ができないか」を設計段階で洗うべきだった。今回はタイトルのステージセレクトだけを見ていて、ポーズメニューのステージ移動を見落とし、実装後に気づいて追加対応した。
- スクリーンショット確認を早い段階(実装直後)に入れたことで、はみ出し・重なりを手戻り小さく潰せた。UI作業は「ロジックのテスト → 早期スクショ → 数値検証」の順が効率的。

## 次回への改善提案
- 制限・権限系の機能(ロック、非活性化)を入れるときは、対象操作に到達できる**全経路を先に列挙**して requirements に書く(今回: タイトルのカード + ポーズのステージ移動の2経路)。
- リザルトランク(S/A/B)導入時は `StageCardModel` にフィールドを足し、カード下部の BEST 行の横にランクを置く(受け皿は用意済み)。
- ポーズメニューのステージ移動パネルのカード化(今回はテキストリスト+ロックのみ)は、需要があれば `stageSelect.ts` のカード生成関数を共通化して対応する。
