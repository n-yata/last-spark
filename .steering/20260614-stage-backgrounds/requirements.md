# 要求内容

## 概要

全6ステージのゲーム本編背景を、単色塗りからストーリー世界観に連動した手続き生成のパララックス背景(グラデーション空＋複数層の遠景シルエット)へ刷新する。アセット追加はせず、既存のプレースホルダ哲学(Graphics による手続き生成)に沿ってコードで描く。

## 背景

- 現状、ゲーム本編の背景は `GameScene.setupCamera()` 内の `cam.setBackgroundColor(this.stage.backgroundColor ?? '#0a0e14')`(`src/scenes/GameScene.ts:430`)による単色のみ。
- 景観・空・奥行き(パララックス)が一切なく、stage4/5/6 の僅かな色違いを除けば全ステージが同じ画面に見える。
- 一方で演出シーン(`CutsceneScene` + SVG 背景)はリッチで、本編との落差が大きい。
- ストーリーは6ステージで明確な世界観を持つ(崩れた都市→立坑の街→収容施設→汚染地帯→ECLIPSE外縁部→支配中枢)。`src/config/story/stageN.ts`(`docs/story.md` が正本)が各ステージの世界観を定義しており、背景で表現する素地がある。
- ストーリーがしっかりしているのに背景に雰囲気がまったく出ていない状態を解消し、6ステージが見ただけで判別でき、世界観の没入感が出る状態にする。

## 実装対象の機能

### 1. 背景描画の共通基盤(パララックス背景レイヤー)

- `GameScene` に、ワールド座標で描く背景レイヤー群を追加する(空グラデーション＋1層以上の遠景シルエット)。
- 遠景は `setScrollFactor(<1)` でカメラスクロールより遅く動かし、パララックス(奥行き)を出す。
- 背景はゲームプレイ要素より背面に描く(地形 depth 0・梯子 depth 5 等より後ろ)。視認性・操作性を損なわない。
- `applyCameraLayout()`(RESIZE/高DPI/ズーム)・ボス戦のカメラ bounds 縮約後も破綻しないこと。
- ステージ定義(`StageData`、`src/config/stage1.ts`)から背景テーマを引けるようにし、新ステージでもデータ追加だけで背景が付く形にする(既存の `backgroundColor` と矛盾しない形で拡張)。

### 2. 各ステージのテーマ別背景(全6ステージ)

ストーリー世界観に対応した配色・モチーフで、6ステージを視覚的に差別化する。各テーマの方向性:

- **stage1 崩れた都市**: 錆・蔓草に覆われた旧市街。くすんだ空＋崩壊したビル群のシルエット。見張られている不穏さ。
- **stage2 立坑の街**: 縦に潜る工業都市。鉄骨・足場・縦坑の骨組み、地下へ降りる暗さ。
- **stage3 収容施設**: 無機質な管理施設。収容区画・監視灯・冷たい人工照明、閉塞感。
- **stage4 汚染地帯**: 環境破壊の現場。淀んだ緑の靄・枯れた構造物(既存 `#151a0c` を基調に活用)。
- **stage5 ECLIPSE外縁部**: 冷たい金属。高密度の機械構造体・青い光(既存 `#0c1119` を基調に活用)。
- **stage6 ECLIPSE支配中枢**: 太陽を遮る影の核。深い闇＋巨大コアの威圧、不吉な発光(既存 `#06080f` を基調に活用)。

## 受け入れ条件

### 背景描画の共通基盤

- [ ] 全6ステージで、起動して走行すると単色でない背景(空グラデーション＋1層以上の遠景シルエット)が表示される。
- [ ] 遠景シルエットがカメラスクロールに対し前景より遅く動く(パララックス)ことを目視で確認できる。
- [ ] 背景がゲームプレイ要素(地形・梯子・敵・プレイヤー・弾・UI)より背面に描かれ、これらの視認性・操作性を損なわない。
- [ ] RESIZE・高DPI・縦/横どちらでもレイアウトが崩れない(`applyCameraLayout` と整合)。
- [ ] ボス戦突入でカメラ bounds がアリーナへ縮約された後も背景が破綻しない。
- [ ] 背景テーマはステージ定義データから引かれ、本番バンドルに開発専用情報が混入しない。

### 各ステージのテーマ別背景

- [ ] stage1〜6 の背景が、それぞれのストーリー世界観に対応した配色・モチーフになっている。
- [ ] 6ステージの背景が互いに視覚的に区別できる(並べて見て同一に見えない)。
- [ ] 既存の `backgroundColor`(stage4/5/6)がベース色として活かされる、または置換しても世界観と矛盾しない。

### 品質

- [ ] `lint` / `typecheck` / `test` / `build` がすべて通る。
- [ ] 既存の E2E プレイスルー(`tests/e2e/play-through/`)が通る。

## 成功指標

- 定性: 6ステージをプレイ(またはスクリーンショット)した際、背景だけで「どのステージか」「どんな世界か」が伝わる。ストーリーの雰囲気が画面に出ている。
- 定量: 既存テスト(unit/integration/e2e)・lint・typecheck・build がすべてグリーンを維持。背景追加による回帰がない。

## スコープ外

以下はこのフェーズでは実装しません:

- 本物の描き込みアート / ステージ用 SVG 画像アセットの作成(手続き生成で表現する)。
- 前景の凝ったアニメーションギミック(過剰な動的演出)。霧の微妙な流れ等を入れるかは設計フェーズで判断(最小限に留める)。
- BGM・効果音などサウンド面の変更。
- 新規ステージの追加、ストーリーテキスト(`src/config/story/`・`docs/story.md`)の変更。
- 演出シーン(`CutsceneScene`)の背景変更。

## 未決事項

実装(設計)フェーズで判断・確認する:

- パララックスの層数(空＋遠景1層の2層構成か、遠景＋中景の3層構成か)。
- 各ステージの正確なカラーパレット値と、シルエットの具体モチーフの作り込み度合い。
- 背景レイヤーの depth/scrollFactor の具体値(地形 depth 0 より背面に置く方式: depth 負値 か、最初に add して相対背面にするか)。
- `StageData` への背景テーマ表現方法(既存 `backgroundColor` を拡張するか、`backgroundTheme` 等の新フィールドを足すか)。
- 背景に軽いアニメーション(ゆらぎ・グロー明滅)を入れるか否か。

## 主要参照コード(設計フェーズ用メモ)

- `src/scenes/GameScene.ts:430` — 現状の単色背景 `cam.setBackgroundColor(...)`。背景レイヤー追加の起点。
- `src/scenes/GameScene.ts:439-451` — `applyCameraLayout()`(zoom/viewport/RESIZE)。背景が整合すべき対象。
- `src/scenes/GameScene.ts:427,502` — カメラ bounds(初期 0,0,width,540 → ボス時アリーナ縮約)。
- `src/scenes/GameScene.ts:326-334` — `tileSprite` 利用例(梯子)。横長背景のタイル敷きに転用可。
- `src/config/stage1.ts` — `StageData` 定義と全6ステージのデータ(`backgroundColor` フィールド)。
- `src/config/balance.ts:384-391` — `STAGE`(width/height/groundY)。ワールド座標系。
- `src/config/dimensions.ts` — `GAME_WIDTH=960` / `GAME_HEIGHT=540`。
- depth 体系: 地形 0 / 梯子 5 / ログ・ケージ 6 / 敵 8 / ボス 9 / プレイヤー 10 / エフェクト 20 / UI 95-101(`setScrollFactor(0)`)。背景はワールド層の最背面に。
- スケール規約: ワールド内オブジェクトは `uiScale` 不適用(論理 px のまま)。背景もワールド座標なら同様にスケール不要。

## 検証方法(実装後)

1. `npm run dev` で起動し、ステージ選択から stage1〜6 を順に開始 → 走行して背景の景観・パララックス・テーマ差を目視確認(スクリーンショット比較)。
2. ボス戦まで進め、カメラ縮約後も背景が破綻しないか確認。
3. ウィンドウリサイズ・縦横切替・高DPI環境で崩れがないか確認。
4. `npm run lint` / `typecheck` / `test` / `build` を全通し。`tests/e2e/play-through/` のプレイスルーが緑であること。
