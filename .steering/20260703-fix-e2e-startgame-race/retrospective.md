本番側は startIntro の遅延コールバックに `ended` ガード(1行)を追加した。

## 学んだこと

**技術的な学び**:
- **Phaser は scene の pause/resume で保持中キーの isDown をリセットする**。Playwright の
  `keyboard.down()` は1回きりのイベントなので、pause を挟むテストでは「押しっぱなし」が
  静かに解除される。長時間の走破テストでは保持キーを毎周再送するのが安全。
- **pause 中のシーンはカメラ更新も止まる**。`transitionTo`(cam.fadeOut → FADE_OUT_COMPLETE →
  scene.start)の途中で pause されると遷移ごとデッドロックする。遅延起動(delayedCall)で
  scene.pause() する処理には、終了状態(`ended`)ガードを入れて競合の芽を摘む。
- **「非存在」を1回見て抜ける待ち方は遅延起動と競合する**。e2e ヘルパーは「出現を待って消化する」
  か「十分な連続観測で不在を確定する」方式にする(今回: 出現待ち5s / 不在確定は実行中1.5s連続)。
- 同一の根因が2つの全く異なる症状(遷移デッドロック / 入力無反応の凍結)として現れた。
  症状ベースで別々に追うより、時系列トレース(50ms間隔のシーン状態記録)で共通原因に到達できた。

**プロセス上の改善点**:
- 調査フェーズでページ内トレーサー(setInterval でシーン状態を記録)と使い捨ての Playwright
  スクリプトを使ったことで、テストランナー越しでは見えない「pause の瞬間」を特定できた。
- E2E_PORT(PR #118 以降)で並行セッションとポートを分離して実行。reuseExistingServer の
  「他人のサーバーを掴む」事故を回避できた。

## 次回への改善提案
- **CI は e2e を workflow_dispatch 限定でしか回していない**(PR CI に含まれない)。今回のような
  競合はマージ後もローカルでしか気づけないため、少なくとも smoke e2e を PR CI に組み込む検討を
  推奨(別作業として)。
- 本体ツリー(メイン worktree)の node_modules が空になっていた。空のまま `npx playwright` を
  叩くと npx キャッシュの別バージョン(1.61.1 ≠ ロック版 1.60.0)に落ちて検証条件が変わる。
  本体ツリーで `npm ci` を実施して復旧すること(未実施。テスト実行は worktree 側で行った)。
- タイミング依存の e2e 失敗を調査するときは、まず「クリーン環境(npm ci + ロック版)で
  単独/並列の両方」を再現条件として押さえると、環境要因と本体要因の切り分けが早い。
