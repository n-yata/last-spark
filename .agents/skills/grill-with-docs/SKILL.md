# grill-with-docs

**説明**
プロダクトや機能のアイデアを、インタビュー形式で徹底的に掘り下げる壁打ちセッションです。
永続ドキュメント(`docs/`)を作成する**前段**として、固まった内容を `docs/ideas/` に成果物として書き出します。

このスキルのゴールは「正式ドキュメントを書くこと」ではありません。
正式ドキュメント(PRD・機能設計書など)を書き始められる状態まで、アイデアを言語化・構造化することがゴールです。

---

# 出力先

壁打ちの成果は `docs/ideas/` 配下に書き出します。

| ファイル | 用途 |
| --- | --- |
| `docs/ideas/initial-requirements.md` | プロダクト/機能の初期要求。`prd` スキルがこのファイルを前提に PRD を生成する。**主たる成果物**。 |
| `docs/ideas/<topic>.md` | 技術調査メモ、検討中の選択肢、アイデアの断片など。自由形式。 |

`docs/ideas/` は `/setup-project` 実行時に自動的に読み込まれます。
ここに書いた内容が、後続の永続ドキュメント作成の入力になります。

---

# 実施内容

## インタビュー形式で徹底的に掘り下げる

アイデアについて、共通理解に到達するまであらゆる観点から質問を行います。

検討すべき分岐ごとに順番に進め、依存関係のある意思決定を一つずつ解消していきます。

各質問について、

* 私からの質問
* 私が推奨する回答

を提示します。

質問は必ず一度に一つだけ行い、回答を待ってから次へ進みます。

### 掘り下げる観点

`initial-requirements.md` を埋められる状態にするため、最低限以下を明らかにします。

* 解決したい課題は何か(なぜ作るのか)
* ターゲットユーザーは誰か
* 実装したい主要機能
* MVP の範囲(最初に作る/作らないの線引き)
* 既存の制約(技術スタック、外部依存、納期など)

---

## コードベースとドキュメントを先に調査する

質問内容がリポジトリの調査によって判断できる場合は、ユーザーへ質問する前にまず自分で確認します。

* 既存の `docs/`(PRD・機能設計書など)
* 既存の `docs/ideas/` の内容(壁打ちの続きの可能性がある)
* 関連する既存実装(`src/` 等)

既に答えがある内容を質問してユーザーの時間を奪わないでください。

---

# セッション中のルール

## あいまいな用語を明確化する

意味が曖昧な言葉や複数の意味を持つ言葉を使った場合は、より正確な言葉を提案します。

例：

> 「アカウント」と言っていますが、「顧客」と「ユーザー」のどちらですか？
> このプロダクトでは両者は異なる概念になりそうです。

---

## 具体的なシナリオで検証する

機能や概念の関係を議論する際は、具体的なシナリオを作って検証します。

境界条件や例外ケースを意図的に作り出し、責務や範囲を明確にします。

---

## 既存ドキュメントとの矛盾を指摘する

ユーザーの説明が、既存の `docs/` や `docs/ideas/` の記述と矛盾する場合は必ず指摘します。

例：

> 既存の PRD では対象を「法人ユーザーのみ」としていますが、今回は個人ユーザーも含める想定でしょうか？

---

# ドキュメント作成ポリシー

ファイルは必要になった時点で作成します。事前作成は禁止です。

## 都度書き出す

意思決定が固まるたびに、その場で `docs/ideas/` のファイルへ追記します。

後でまとめて書こうとせず、確定した内容から都度記録してください。

## docs/ideas/initial-requirements.md

最初の要求(課題・ターゲット・主要機能のいずれか)が確定したタイミングで作成します。

PRD 作成の前提となるため、以下の見出しを揃えておきます。

```markdown
# 初期要求

## 解決したい課題

## ターゲットユーザー

## 主要機能

## MVP の範囲

## 制約・前提
```

## docs/ideas/<topic>.md

技術調査メモや検討中の選択肢など、要求とは別に残しておきたい内容は
トピックごとの自由形式ファイルとして書き出します。

## 自由形式を尊重する

`docs/ideas/` は下書きの場です。正式ドキュメントのような厳密な構造は求めません。
過度な体裁は整えず、思考の過程・未確定の論点・却下した案も残して構いません。

---

# セッションの終わり方

壁打ちが一区切りしたら、次のいずれかをユーザーに案内します。

* PRD を作るなら `prd` スキル(`docs/ideas/initial-requirements.md` を入力にする)
* 永続ドキュメント一式を作るなら `/setup-project`(`docs/ideas/` を読み込む)

未解決のまま残した論点があれば、`docs/ideas/` 内に明記してから終えてください。
