# 技術仕様書 (Architecture Design Document)

> `docs/product-requirements.md`(PRD)と `docs/functional-design.md`(機能設計書)で定義した要件を、技術的にどう構築・配信するかを定義する。対象は「Phaser 3 製の横スクロール2Dアクション(全6ステージ+全6ボス)」を「スマホ向け PWA(完全オフライン)」として配信する構成。

## アーキテクチャパターン

### レイヤードアーキテクチャ(クライアント内)

サーバを持たない単一クライアントアプリのため、レイヤー分離は「ブラウザ内」で完結する。依存方向は上から下への一方向に保つ。

```
┌─────────────────────────────────────┐
│  Scene レイヤー(UI/状態)            │ ← 画面・ゲーム状態の管理、入力受付
├─────────────────────────────────────┤
│  System レイヤー(ゲームロジック)    │ ← 入力解釈・戦闘判定・敵出現・ボスAI
├─────────────────────────────────────┤
│  Entity レイヤー(ドメインオブジェクト)│ ← Player/Enemy/Boss/Projectile の状態と振る舞い
├─────────────────────────────────────┤
│  Persistence レイヤー(永続化)       │ ← SaveManager → localStorage
└─────────────────────────────────────┘
```

#### Scene レイヤー
- **責務**: 画面/ゲーム状態の管理(Boot/Preload/Title/Game/UI/Cutscene/GameOver/Clear/Orientation)、入力イベントの受付、シーン遷移。`CutsceneScene` は演出(交互テキスト+ト書き)を再生する独立シーン。
- **許可される操作**: System レイヤーの利用、Persistence の呼び出し(セーブ/ロード)。
- **禁止される操作**: 物理衝突やボスAIの計算ロジックを直接持つこと(System に委譲)。

> **補足(HUD/演出の描画分離)**: `UIScene` は `GameScene` と並行起動する独立シーンで、専用カメラを持つ。プレイ画面(`GameScene` カメラ)に適用するポストFX(色調補正/ブルーム/ビネット、`config/graphicsQuality.ts`)は `UIScene` のカメラへは波及しないため、HUD・ストーリーテキストは FX の影響を受けず視認性を保てる。シーンを分離していること自体が、本体カメラだけに後処理を掛けられる構造的な利点になっている。

#### System レイヤー
- **責務**: 入力解釈(InputController)、衝突/ダメージ(CombatSystem)、敵出現(SpawnSystem)、ボス行動抽選などのゲームロジック。
- **許可される操作**: Entity の状態更新、Entity への振る舞い指示。
- **禁止される操作**: Scene への逆依存(コールバック/イベントで通知する)。

#### Entity レイヤー
- **責務**: Player/Enemy/Boss/Projectile が自身の状態(HP・向き等)と最小限の振る舞い(`update`)を持つ。
- **許可される操作**: 自身の物理ボディ操作、自状態の更新。
- **禁止される操作**: 他 Entity の探索や全体ロジック(System が統括)。

> **補足(ボス系統のコンフィグ駆動継承)**: `Boss`(接地型・既定)を基底に、`FlyingBoss`(飛行/浮遊型・stage2)・`WardenBoss`(重装型・stage3)・`PurifierBoss`(浄化型・stage4)・`CoreBoss`(ラスボス・stage6)・`ShadowRayBoss`(hard mode 専用裏ボス・stage6)がサブクラス化されている。`FlyingBoss` はさらに `EnvoyBoss`(高速型・stage5 の使者)にサブクラス化され、滞空高度・急降下等の固有値だけを差し替える。各系統は `BossConfig`(コンストラクタ引数)で重力有無・パラメータをコンフィグ駆動にしつつ、`beginNextAction` / `executeAction` / `clampToArena` / `updateRig` の一部のみを上書きする。ボスは任意で `shieldHp`(被弾を吸収するシールド値)を持ち、`resolveBossShieldHit`(純粋関数)でシールド優先の被弾処理を解決する。
>
> **補足(Beam エンティティ)**: 強化バスターモード(`GameSettings.busterMode`)向けに `Beam` を `Projectile` とは別の Entity として追加している。通常/チャージ弾(`Projectile`、`kind: 'normal' | 'charged' | 'missile' | 'lance'`)が単発の飛翔体であるのに対し、`Beam` は発射中は追従する持続チャージビームであり、当たり判定・寿命の扱いが異なるため意図的に別クラスとした。

> **補足(見た目リグの分離)**: キャラの物理(`Phaser.Physics.Arcade.Sprite`)と見た目を分離している。物理エンティティは自スプライトを非表示にし、頭・胴・腕・脚のパーツを `Container` で関節化した `CharacterRig`(`entities/CharacterRig.ts`)へ表示を委譲する。各パーツの変位は Phaser 非依存の純粋関数 `systems/rigAnimation.ts`(歩行スイング・スクワッシュ&ストレッチ・発射反動・被弾のけぞり)で算出し、系統別の構成は `config/characterRig.ts` に集約する。テクスチャは外部素材を使わず `PreloadScene` が同定義から手続き生成する(知財・コンプライアンス方針に準拠)。将来の外部スプライトシート移行時も、物理に触れず `CharacterRig` 差し替えで対応できる。
>
> **補足(アートディレクション方針)**: プレイ画面(背景/敵/地形/プレイヤー)は手続き生成の画風に統一する。厚塗り一枚絵(外部生成のキービジュアル)は、タイトル画面・カットシーンのような静止画的演出に限定して使う。過去にプレイヤーキャラのみ外部生成の横向きカットアウト・リグ(`entities/SpriteRig.ts` / `config/raySprite.ts`)へ差し替えた際、敵・地形の手続き画風から浮いて世界観の統一感を損ねたため、手続きの `CharacterRig` へ差し戻した経緯がある(`SpriteRig`/`raySprite` は将来の再利用のため無効化した状態でリポジトリに残置)。プレイ画面のアートを底上げする場合は、一部キャラだけでなく敵/地形/背景を同じ画風へ一括で引き上げてから適用する。
>
> **補足(ワールド生成ロジックのUI流用時の注意)**: ステージ選択カードのミニプレビュー等、`GameScene` の手続き背景生成ロジックを UI 側で流用する場合、`Phaser.GameObjects.Graphics` の直描画はカメラ/コンテナのクリップが自動では効かない。ワールド側では画面(カメラ)がクリップの役目を果たしているため、UI に流用する際は描画側で明示的にクリップ領域を設ける必要がある。

#### Persistence レイヤー
- **責務**: `SaveData` の読み書き、既定値生成、バージョン検証。
- **許可される操作**: `localStorage` へのアクセス。
- **禁止される操作**: ゲームロジックの実装、外部ネットワーク通信。

> **依存方向**: Scene → System → Entity / Scene → Persistence。逆方向(Entity → Scene 等)は禁止。System から Scene への通知はイベント/コールバックで行い、直接参照しない。
>
> **補足(純粋関数モジュールの扱い)**: `systems/` 内の Phaser 非依存の純粋関数モジュール(camelCase ファイル: `bossAi.ts` / `shot.ts` / `combatRules.ts` / `playerMovement.ts` / `hudFx.ts` 等)は、状態を持たない最下位ロジックとして `config/` / `types/` と同等に扱い、Entity および UI コンポーネントからの参照を許可する。Entity/UI が依存してはならないのは状態・副作用を持つ **System クラス**(`InputController` / `CombatSystem` / `SpawnSystem` 等の `*System`/`*Controller`)のみである。これにより、戦闘・抽選・移動・HUD 演出などのコアロジックを純粋関数で共有しつつ、テスト容易性(Phaser 非依存)を保つ。

### チューニング値の集中管理
難易度・手触りに関わる数値(速度・ジャンプ力・ダメージ・チャージしきい値・ボスHP 等)は `src/config/` の定数モジュールに集約する。本番コードへのマジックナンバー埋め込み・`if (testMode)` 等のテスト用分岐は禁止。

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| クリア状況/最速タイム | localStorage | JSON 文字列 | 完全オフライン方針。端末内で完結しサーバ不要 |
| ユーザー設定(ミュート/音量/難易度) | localStorage | JSON 文字列 | 同上。再訪時に設定を復元 |
| ステージ/敵配置データ(全6ステージ) | ソースコード同梱 | コード定義(`config/stages.ts` の `StageData`、`config/story/stage1〜6.ts` のストーリーテキスト) | 読み取り専用のゲームデータ。Tiled 等の外部マップは使わずコードで定義し、ビルドに同梱・SW でキャッシュ |

- 保存キーは名前空間付き `lastspark:save`。
- 読み込み時に型・`version` を検証し、不一致/破損は既定値にフォールバック(進行不能を防ぐ)。

### バックアップ戦略

- **方針**: サーバを持たないため、端末外バックアップは行わない(スコープ外)。
- **データ堅牢性**: セーブは「クリア確定時」「設定変更時」のみの低頻度書き込みとし、破損リスクを最小化。読み込み失敗時は既定値で起動し、ゲームは継続可能(進捗が無いだけ)。
- **将来**: オンライン要素導入時にサーバ同期を検討(公開後の拡張候補)。

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| 入力(タッチ)→キャラ反応 | 1〜2フレーム以内(60fps 換算で約 16〜33ms) | ミドルレンジ実機(Android Chrome / iOS Safari) |
| 2回目以降の起動→タイトル表示 | SW キャッシュ利用で高速起動(確定目標は運用フェーズで計測) | ミドルレンジ実機 |
| ゲームプレイ中のフレームレート | 平均 60fps 維持(ボス戦含む。具体的な下限は運用フェーズで計測) | ミドルレンジ実機 |

- 測定方法: Chrome DevTools / Performance パネルでフレーム描画時間と起動時間を計測する。フレーム落ちが顕著な箇所はプロファイルして最適化。
- 起動時間とフレームレート下限の確定目標値は現時点では未設定。実機での基準端末選定と計測手順の確定を含め、運用フェーズで計測ガイドとして整備する。
- **グラフィック品質モード**(`config/graphicsQuality.ts`・`GameSettings.graphicsFx`): プレイ画面のポストFX(色調補正/ブルーム/ビネット)を AUTO/HIGH/OFF で切替可能にし、高密度端末(生 devicePixelRatio が `BLOOM_MAX_DPR` を超える機種)では AUTO 時に負荷の高い bloom のみ自動的に無効化して 60fps を優先する。Canvas フォールバック(非 WebGL)時はモードに関わらず全FX無効。

### リソース使用量

| リソース | 上限/目安 | 理由 |
|---------|------|------|
| 初回ダウンロード総量 | できる限り小さく(具体バジェットは運用フェーズで計測) | モバイル回線/初回ロードの体感。外部素材を持たず手続き生成のため肥大化要因が少ない(主因は Phaser ランタイム) |
| 実行時メモリ | ミドルレンジ端末で安定動作する範囲(具体上限は運用フェーズで計測) | 弾/エフェクトのプール再利用で増加を抑制 |
| 描画ドローコール | 最小化 | 手続き生成テクスチャをバッチ描画 |

> 初回ダウンロード総量の具体的なバジェット(端末別の上限・計測手順)は、実測に基づき運用フェーズの計測ガイドで確定する。

## セキュリティアーキテクチャ

### データ保護

- **外部通信なし**: サーバ通信を行わず、外部送信による情報漏洩経路を持たない。
- **機密情報管理**: API エンドポイント/キー/トークン等を保持しない(該当機能なし)。万一将来導入する場合も、ソース・ドキュメントへのハードコードを禁止し、ビルド時環境変数(`VITE_*`)で注入する。
- **localStorage**: 自端末内データのみを扱う。暗号化は行わない(機密性のあるデータを保存しないため)。

### 入力検証

- **タッチ入力**: InputController が抽象操作に正規化。不正なポインタ状態でも例外を出さず安全側に扱う。
- **セーブデータ検証**: localStorage 読み込み値は型/`version`/値域を検証し、不正値は既定にフォールバック(改ざんで進行不能にならない)。
- **エラーハンドリング**: 実行時例外は当該シーンを安全に停止しタイトルへ復帰。スタックトレース等の内部情報をユーザーに露出しない。

### 知的財産・コンプライアンス
- キャラ・スプライト・名称・音楽はすべてオリジナル、または CC0 等の商用利用可ライセンス素材に限定。既存作品(ロックマン等)の素材・名称・固有名は使用しない。
- 使用素材のライセンス出典をリポジトリ内(例: `CREDITS.md` / アセット README)に記録する。

## スケーラビリティ設計

### データ増加への対応
- 全6ステージ分のステージデータ・敵配置・ストーリーテキスト(`config/stages.ts` / `config/story/stage1〜6.ts`)をコード定義で保持。永続データ(セーブ)自体はクリア状況/設定/周回数のみで小規模だが、`clearedStages` / `bestTimeMs` / `bestRank` はステージ数に比例して増える構造(現状6ステージ分)。データ量起因の性能劣化は想定しない。
- アセットは増えるため、ステージ追加時もアトラス/圧縮/遅延ロード(必要シーンでのみロード)で初回ロードを抑える方針。

### 機能拡張性
- **ステージ追加**: ステージは `config/stages.ts` の `STAGES` テーブルに `StageData`(地形・敵配置・ボストリガー等)をコード定義で追加できる構造。`SpawnSystem` は `stageId` で `getStageData` を引く。全6ステージで一区切りだが、7ステージ目以降を追加する場合も同じ構造で拡張できる。
- **ボス系統の拡張**: `Boss` を基底に `BossConfig` で重力・パラメータをコンフィグ駆動にしつつ、系統差分が大きい場合はサブクラス化(`FlyingBoss` → `EnvoyBoss` の例)で対応する構造が確立済み。任意の `shieldHp` によるシールド機構も既に一般化されている。
- **ショット種別の拡張**: `Projectile` の `kind` は `'normal' | 'charged' | 'missile' | 'lance'` へ拡張済み。持続系の攻撃(強化バスターのチャージビーム)は当たり判定・寿命の性質が異なるため `Beam` という別 Entity として追加した。今後ボスごとの弱点武器システムを導入する場合も、この `kind` 拡張と `Beam` 分離の前例に倣う。
- **オンライン/ランキング(公開後の拡張候補)**: Persistence レイヤーの背後を localStorage からサーバ同期に差し替え可能なよう、SaveManager をインターフェースとして抽象化しておく。

## テスト戦略

### ユニットテスト
- **フレームワーク**: Vitest
- **対象**: ボス行動抽選(`pickNextAction`)、チャージ判定の境界、SaveManager(既定値/バージョン不一致/例外時)、ダメージ・無敵時間ロジック。純粋ロジックは Phaser 非依存に切り出してテスト可能にする。
- **カバレッジ目標**: コアロジック(config/system の純粋関数)で 80% 以上を目安。カバレッジ値より実質的な検証を優先。

### 統合テスト
- **方法**: Vitest 上で System ＋ Entity を結合し、入力→移動/発射、衝突→HP 減少→撃破/ゲームオーバー遷移を検証。
- **対象**: InputController〜Player〜CombatSystem の連携。

### E2Eテスト
- **ツール**: Playwright
- **シナリオ**: タイトル→スタート→導入区間突破→ボス撃破→クリア→タイトル。画面回転(縦持ち案内)。リロード後のクリア状況保持。

## 技術的制約

### 環境要件
- **実行環境(プレイヤー)**: モバイルの主要ブラウザ(iOS Safari / Android Chrome)。横向き(ランドスケープ)・両手持ち前提。
- **開発環境**: devcontainer(Node.js v24 / npm)。
- **必要な外部依存**: なし(プレイ時はオフラインで完結)。

### パフォーマンス制約
- モバイル GPU/CPU 性能を前提に、物理は Arcade Physics に限定。重い後処理シェーダは多用しない。
- iOS Safari のオーディオ自動再生制約により、サウンドは初回タップ後に解放する。

### セキュリティ制約
- 外部送信を行わない(完全オフライン)。
- 機密情報をソース/ドキュメントに含めない。
- 第三者の知的財産を使用しない。

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| phaser | ゲームエンジン | `^`(マイナーまで自動)。メジャー更新は手動検証 |
| vite | ビルド/開発サーバ | `^` |
| vite-plugin-pwa | PWA 化 | `^` |
| vitest | テスト | `^`(devDependencies) |
| @playwright/test | E2E | `^`(devDependencies) |
| typescript | 型/コンパイル | `~`(パッチのみ自動、devDependencies) |
| eslint / prettier | 静的解析/整形 | `^`(devDependencies) |

**方針**:
- ランタイム依存は `^` でマイナーまで許容し、`package-lock.json` で厳密に固定。
- 破壊的変更リスクの高いものはメジャー更新時に手動検証。
- 型ツールはパッチのみ自動(`~`)で安定性を優先。
